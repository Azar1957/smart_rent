# Smart Rent — Техническое задание на фронтенд

> **Статус документа.** Действующее ТЗ для веб-фронтенда smart rent и
> для будущего мобильного приложения (iOS / Android), которое должно
> предоставлять тот же функционал и работать с тем же REST API.
>
> **Аудитория документа.** Frontend-инженер, дизайнер UX/UI, QA,
> мобильные инженеры, продакт-менеджер.
>
> **Цель.** Описать, как устроен личный кабинет арендатора и
> арендодателя, какие сценарии в каждой роли, какие компоненты и
> ограничения, и как делать UI адаптивным под телефон / планшет /
> десктоп так, чтобы тот же набор экранов потом легко превратить в
> мобильное приложение.

---

## 0. Контекст и стек

- **Web-фронт.** Next.js 14 (App Router), TypeScript, Tailwind CSS,
  zustand для глобального стейта, SWR для запросов, lucide-react для
  иконок, next/image для изображений.
- **Backend.** InterSystems IRIS 2026.1, REST API под `/api/smartrent/v1`
  (на фронте проксируется через `/api/iris/*`).
- **Дизайн-система.** «Hyer Aviation luxury»: палитра obsidian/canvas/
  mistx + единственный акцент Desert Sienna (`#bc7155`); шрифт-substitute
  Manrope (близкий к HelveticaNowDisplay); pill-кнопки 1000px; крупные
  display-заголовки. Полная карта токенов — в `tailwind.config.js`,
  `globals.css` и в `Structure.md`.

## 1. Роли и роутинг

> **Что должен помнить разработчик.** Авторизация — bearer-токен в
> `localStorage` (smartrent-auth). После логина пользователь
> перенаправляется в `/dashboard`. Дашборд **разный для тенанта и
> арендодателя** и не имеет общей плоской таблицы.

### 1.1 Карта роутов

| Роут | Доступ | Назначение |
|---|---|---|
| `/` | публично | главная: hero + витрина листингов |
| `/listings/[id]` | публично | детальная карточка объявления (галерея, sticky-цена, бронирование) |
| `/login`, `/register` | публично | авторизация |
| `/dashboard` | auth | личный кабинет, ветвится по роли |
| `/payments` | auth | хаб платежей: 2 карточки → rent / utilities |
| `/payments/rent` | auth | история rent + (для тенанта) кнопка «Оплатить» |
| `/payments/utilities` | auth | история коммунальных + (для тенанта) кнопка «Оплатить» |
| `/properties`, `/properties/[id]` | auth (`landlord`/`admin`) | управление объектами и сегментами |
| `/bookings` | auth | список заявок (для landlord — с действиями approve/reject) |

> **Защита роута.** Любой `/dashboard*`, `/payments*`, `/bookings*`,
> `/properties*` требует токен. Если токена нет — редирект на `/login`
> через `useEffect` + `router.replace`.

### 1.2 Роли

- `admin` — администратор платформы; внешний интерфейс совпадает с
  ролью `landlord`, плюс служебные тулзы (на текущем этапе не входят
  в видимый scope).
- `landlord` — арендодатель.
- `tenant` — арендатор.

## 2. Личный кабинет арендатора

> **Главный принцип.** После входа арендатор видит, что у него
> арендуется, ближайший платёж и быстрые действия «Оплатить аренду /
> коммуналку». Дашборд — не дешбоард-простыня с цифрами, а полезная
> рабочая поверхность.

### 2.1 Главная страница `/dashboard` (роль = tenant)

Структура сверху вниз:

1. **Шапка профиля.** Заголовок «Здравствуйте, {Имя}», email и бейдж
   роли. Справа — две CTA-кнопки: `Browse stays` (outline) и
   `Pay rent` (primary sienna).
2. **Сводные карточки** (3 шт., колонками на md+, стопкой на mobile):
   - `Активные аренды` — счётчик;
   - `Всего в месяц` — сумма ежемесячных платежей в выбранной валюте;
   - `Ближайший платёж` — сумма + дата (или «Платежей нет»).
3. **«Мои аренды»** — карточки сданных мне сегментов:
   - фото 16:9, тип (квартира / комната), заголовок, адрес,
     цена/мес, депозит, коммуналка/мес, период действия договора;
   - вся карточка кликабельна → `/listings/{listingId}` (если в
     каталоге есть совпадающее демо-объявление, иначе → на историю
     платежей);
   - две доп. кнопки в подвале: `Pay rent` и `Pay utilities`.
4. **Платежи** — два больших action-card:
   - `Оплата аренды` → `/payments/rent`;
   - `Оплата коммуналки` → `/payments/utilities`.
5. **Подборка новых предложений** (recommendations) — 4 листинга из
   каталога, исключая те, что уже арендуются. Кликабельны на
   `/listings/[id]`.

### 2.2 Страницы платежей

**`/payments` (хаб)** — 2 крупных action-card:
- `Аренда` → `/payments/rent`;
- `Коммуналка` → `/payments/utilities`.

**`/payments/rent` и `/payments/utilities`** — таблица платежей:

| Поле | Tenant | Landlord |
|---|---|---|
| Месяц | да | да |
| Описание (`Rent for YYYY-MM` / `Utilities (kind) for YYYY-MM`) | да | да |
| Аренда (адрес+метка) | да | да |
| Сумма (в текущей валюте) | да | да |
| Срок оплаты | да | да |
| Статус (paid / pending / overdue) — иконка + цвет | да | да |
| Дата фактической оплаты (для paid) | да | да |
| Кнопка `Оплатить` | да (для не-paid) | нет |

Сверху таблицы — два сводных блока: «Всего оплачено» (нейтральный
цвет) и «К оплате» (если > 0 — sienna). На мобильном — карточки
вместо таблицы.

> **Параметр `?rental=<id>`** в обоих маршрутах фильтрует таблицу по
> конкретному сегменту. Используется со страницы дашборда арендодателя,
> чтобы провалиться к платежам именно того арендатора.

### 2.3 Сценарий «Оплатить»

1. Тенант жмёт `Оплатить` в строке таблицы или CTA на дашборде.
2. Открывается **PayModal**: краткая шапка с описанием платежа, плашка
   с суммой, поля для номера карты / срока / CVC и кнопка `Оплатить`.
3. Имитация запроса 600 ms → success-вид c галочкой; запись локально
   помечается как `paid` (без обращения к серверу — это демо).
4. В реальной интеграции — переход в Stripe Checkout по 302, callback
   `/payments/:id/pay` обновит статус в IRIS.

## 3. Личный кабинет арендодателя

> **Главный принцип.** Арендодатель видит свои объекты в одной плитке,
> агрегаты по сборам и быстрый переход к истории по каждому
> объекту/арендатору. Никакой мешанины с витриной — у него рабочее
> место.

### 3.1 Главная страница `/dashboard` (роль = landlord/admin)

1. **Шапка профиля** — то же, что у тенанта; CTA-кнопка справа —
   `Manage properties` (sienna).
2. **Сводные карточки** (4 шт.):
   - `Сданные единицы` — кол-во активных сегментов;
   - `Доход в месяц` — суммарная месячная плата;
   - `Собранная аренда` — общая сумма всех paid rent-платежей;
   - `Собранная коммуналка` — общая сумма всех paid utility-платежей.
3. **«Мои сданные объекты»** — список объектов, сгруппированных по
   адресу. Каждая группа — горизонтальная карточка:
   - слева большое фото 16:10 (на md+);
   - справа: адрес, заголовок, мини-параметры (`Tenants`,
     `Monthly total`, `Tenants list`);
   - под параметрами — список сегментов: для каждого арендатора
     отдельная строка с метаданными (тип, площадь, email тенанта,
     срок), сводкой по последним платежам (`Last rent`, `Next rent`,
     `Last utility`) и двумя кнопками `Rent history` / `Utility
     history`, которые ведут на `/payments/rent?rental=...` и
     `/payments/utilities?rental=...`.

> **Расширение в будущем.** Когда появятся живые данные через
> `/dashboard/landlord` (есть в IRIS), сюда же должны переехать
> график доходов по месяцам и блок «Просрочки».

### 3.2 История платежей

Те же `/payments/rent` и `/payments/utilities`. Для арендодателя
кнопки оплаты не показываются (он только смотрит). Через `?rental=`
из дашборда фильтруется по конкретному сегменту/арендатору.

## 4. Архитектура клиентского стейта

### 4.1 Глобальный стейт (`zustand` + `persist`)

`useAuth` (`frontend/src/lib/store.ts`):

```ts
{
  token: string | null;
  user: User | null;
  locale: 'en' | 'es' | 'ru';
  currency: 'EUR' | 'USD' | 'GBP' | 'CHF' | 'KZT' | 'AED';
}
```

Persist в `localStorage` под ключом `smartrent-auth`. Сохраняем токен
и валюту/язык — чтобы при перезагрузке страницы не сбрасывать.

### 4.2 i18n

Тонкий wrapper `useT()` в `frontend/src/lib/i18n.ts`. Cловари — в
`frontend/src/i18n/dictionaries.ts`, ровно три локали: `en`, `es`,
`ru`. Тип `Dictionary` строго типизирован, любая новая строка
обязана появиться в трёх локалях, иначе TS-ошибка на сборке.

> **Правило.** Никаких **strings-as-values** в коде компонентов. Всё
> читаемое пользователем приходит из `t.*`. Это критично и для
> мобилки — на ней мы возьмём те же словари.

### 4.3 Локализованные демо-данные

Демо-данные витрины (`frontend/src/data/listings.ts`) и активных
аренд (`frontend/src/data/demoRentals.ts`) хранят `LocalizedText
= { en, es, ru }` для всех текстовых полей и читаются через
`pickLocale(text, locale)` / `readLocalized(text, locale)`. Это
гарантирует мгновенное переключение языка без обращения к серверу.

### 4.4 Валюта

Все цены данных хранятся в **EUR**. На вывод используется
`formatPrice(amountEur, currency)` из `frontend/src/lib/currency.ts`,
который через `Intl.NumberFormat` приводит к выбранной валюте по
индикативному курсу. Это применимо и к мобильному приложению —
логика лежит в чистой ts-функции.

### 4.5 SWR / API-клиент

`frontend/src/lib/api.ts` — тонкий wrapper над `fetch`:
- автоматический `Bearer`-заголовок,
- разбор `{ error: '...' }` в `Error.message`,
- `cache: 'no-store'` (всегда свежие данные),
- единая база (`/api/iris` в браузере, `IRIS_API_BASE` в SSR).

Любой запрос с фронта оформляется через SWR с key=`[path, token]`,
чтобы при logout кэш самоинвалидировался.

## 5. Дизайн-система и компоненты

### 5.1 Базовые компоненты (на странице — не на бэке)

- `Logo` (icon / mark / lockup, tone dark/light) — единый брендовый
  лого; используется в шапке, футере и auth-страницах.
- `Header` — sticky-канвас с навигацией, переключателями валюты/языка,
  CTA «Become a host», бургер для мобайла.
- `ListingCard` — карточка объявления для витрины.
- `BookingModal` (внутри `/listings/[id]`) — заявка на бронирование.
- `TenantDashboard`, `LandlordDashboard` — ролевые композиции
  дашборда.
- `PaymentsTable` — общая таблица/карточки платежей; принимает
  пропсы `rentals`, `filterKind: 'rent' | 'utility'`, `role` (тенанту
  показывает кнопку `Pay`, арендодателю — нет).
- `PayModal` — модалка фейковой оплаты карточкой (внутри
  `PaymentsTable`).
- `Stat`, `Summary`, `StatusBadge` — мелкие визуальные кирпичики.

### 5.2 Адаптивность и breakpoints

| Брейкпойнт | Поведение |
|---|---|
| < 640px (sm) | один столбец, мобильные карточки вместо таблиц, бургер-меню в шапке, sticky-блок цены под основным контентом |
| 640–1024px (md) | две-три колонки в сетках, частично таблицы |
| 1024px+ (lg) | полноценные сетки 3–4 колонки, sticky-блок цены справа от описания |
| 1280px+ (xl) | 4 листинга в строке, расширенные блоки дашборда |

Контент шире `max-w-page = 1760px` не уходит. Все текст-сетки
обнуляют горизонтальный паддинг до `px-6` на мобильном.

### 5.3 Доступность

- Все интерактивные элементы — `<button>`/`<a>`, не `<div onClick>`.
- Все иконки сопровождаются текстом или `aria-label`.
- Контраст obsidian/canvas — 19.7:1 (AAA по WCAG).
- Модалки — `role="dialog" aria-modal="true"`, закрытие по фону / `X`.

## 6. Совместимость с будущим мобильным приложением

> **Ключевая идея.** Мобильное приложение (React Native / Expo)
> разделит с веб-фронтом всё, кроме DOM-рендеринга. Цель — выделить
> «portable core» уже на этом этапе.

### 6.1 Что переиспользуем 1:1

- `frontend/src/lib/api.ts` — клиент REST, не зависит от DOM.
- `frontend/src/lib/currency.ts` — pure functions.
- `frontend/src/i18n/dictionaries.ts` — pure data.
- `frontend/src/data/listings.ts`, `demoRentals.ts` — pure data.
- `frontend/src/lib/store.ts` (zustand) — работает в RN без правок.
- Типы (`User`, `LocalizedText`, `DemoListing`, `DemoRental`,
  `DemoPayment`).

### 6.2 Что нужно подменить на мобилке

- Компоненты разметки (`<div>` → `<View>`, `<img>` → `<Image>` из
  Expo, `<a>` → `<Link>` из expo-router) — в отдельной библиотеке
  компонентов.
- next/font → System fonts + Manrope из Expo Google Fonts.
- next/image → expo-image.
- Модалки — `react-native-modal` или нативный bottom-sheet.

### 6.3 Что НЕ дублировать

- Бизнес-логику: формирование описаний платежей, проверка пересечений
  бронирований, расчёт коммуналки — это уже живёт на бэкенде в IRIS.
- Хэширование паролей — на сервере (`SHAHash(512, ...)`); мобилка
  отправляет plain через TLS.

### 6.4 Карта экранов в мобильном приложении

| Web-роут | Mobile-screen |
|---|---|
| `/` | `Home` (вкладка `Stays`) |
| `/listings/[id]` | `ListingDetail` (modal/stack) |
| `/login`, `/register` | `Login`, `Register` (stack) |
| `/dashboard` | `Dashboard` (вкладка `Profile`) |
| `/payments`, `/payments/rent`, `/payments/utilities` | `PaymentsHub`, `RentPayments`, `UtilityPayments` (stack под `Profile`) |
| `/properties`, `/properties/[id]` | `MyProperties`, `PropertyDetail` (только landlord, stack под `Profile`) |
| `/bookings` | `Bookings` (вкладка `Inbox`) |

Bottom-tab bar в RN: `Stays / Inbox / Profile`.

## 7. Контракт с IRIS (минимальный)

> **На случай, если бэк временно недоступен** — фронт должен
> деградировать «мягко»: дашборд показывает демо-данные из
> `frontend/src/data/demoRentals.ts`, витрина — из
> `frontend/src/data/listings.ts`. Это уже реализовано.

### 7.1 Эндпоинты, обязательные для боевого запуска

- `POST /auth/login` → `{ token, user }`. Должен вернуть `user.role` и
  `user.language`, чтобы фронт сразу выбрал нужную локаль.
- `GET /auth/me` — для восстановления сессии после reload.
- `GET /leases?role=tenant|landlord` — список активных аренд с
  расширенными полями (фото, заголовок, адрес — в идеале уже
  локализованные через `Accept-Language`).
- `GET /payments?kind=rent|utility&role=tenant|landlord` — история
  платежей, отсортированная по `dueDate desc`.
- `POST /payments/:id/pay` — фиксация оплаты (или 302 в Stripe).

### 7.2 Что добавить со стороны IRIS

> Эти изменения мы намеренно не делаем сейчас (вне scope этого
> ТЗ-документа), но фиксируем как backlog:

- В `Lease.ToJSON` отдавать `segment.title`, `segment.area`,
  `property.title`, `property.address`, `property.city`,
  `property.country`, главное `photo` (когда у нас будет S3-хранилище
  фотографий).
- Для `/payments` — параметр `kind` со значениями
  `rent`/`utility`/`deposit`, чтобы не фильтровать на клиенте.
- В `/dashboard/tenant` отдавать массив активных leases с теми же
  расширенными полями + список ближайших платежей; в
  `/dashboard/landlord` — сегментированный список объектов с
  агрегатами по платежам.

## 8. Acceptance criteria (что считается готовым)

1. **Smoke тенант.** Логин под `noah.fischer@smartrent.demo /
   Noah12345!` → попадает на дашборд, видит карточку Berlin R1,
   ближайший платёж, списки. Кнопка `Оплатить` открывает PayModal,
   успешная имитация — статус строки меняется на `paid`.
2. **Smoke арендодатель.** Логин под `lukas.berger@smartrent.demo /
   Lukas12345!` → видит группу «Berlin / Linienstraße 44» с тремя
   арендаторами; клик «Rent history» переводит на `/payments/rent` с
   фильтром `?rental=...` и таблицей этого сегмента.
3. **i18n.** Переключение языка в шапке мгновенно меняет все надписи
   личного кабинета и витрины. Никаких «hardcoded strings» в
   компонентах.
4. **Адаптивность.** На viewport 360×640 (iPhone SE) ни одна страница
   не имеет горизонтального скролла; таблица платежей превращается в
   карточки.
5. **Сборка.** `npm run build` проходит зелёным (`✓ Compiled
   successfully`, 0 ошибок типов, 0 ESLint-ошибок).
6. **Без RU-данных.** Глобальный grep `Россия|Невский|Москва|RUB` в
   реальных данных — пусто.
7. **Новые экраны включены в роутинг.** `/dashboard`, `/payments`,
   `/payments/rent`, `/payments/utilities` — все four зеленеют в
   Build-output.

## 9. Roadmap (вне scope текущего этапа)

> **Фиксируем здесь**, чтобы не забыть и чтобы новый разработчик
> увидел общий вектор.

- Полноценный календарь занятости сегмента в `/listings/[id]`.
- Чат с арендодателем (real-time через WebSocket из IRIS).
- Реальная интеграция Stripe Checkout + webhook на `/payments/:id/pay`.
- Загрузка фотографий объекта арендодателем и хранение в S3.
- Переключение тёмной темы.
- Push-уведомления (через FCM/APNs) — для мобильного приложения.
- Аналитика арендодателя: график доходов, прогноз cash-flow,
  KPI «занятость / средний срок».
